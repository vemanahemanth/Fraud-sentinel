1. ABSTRACT
================================================================================

Financial fraud continues to escalate as digital transactions grow worldwide,
costing institutions billions annually and eroding consumer trust. This project,
Fraud Sentinel, is a full-stack real-time fraud detection and case management
platform designed to monitor, score, and act on suspicious financial
transactions. The system combines a PostgreSQL relational database (hosted on
Neon) for structured transactional and compliance data, a MongoDB Atlas document
store for enriched fraud-scoring audit trails, and Redis (Upstash) as a caching
and velocity-tracking layer. The backend is built with Node.js/Express and
features a custom fraud engine that evaluates transactions in real-time using
rule-based scoring and a simulated ML anomaly detection model (Isolation Forest).
The frontend is a modern React dashboard (built with Vite and TailwindCSS)
offering 12+ interactive views including transaction monitoring, alert
management, case investigation, SAR filing, and analytics with geolocation
heatmaps. The platform achieves sub-100ms scoring response times and provides
end-to-end fraud lifecycle management from detection to regulatory reporting.


================================================================================
2. PROBLEM STATEMENT
================================================================================

Financial institutions face an ever-increasing volume of digital transactions
and struggle to identify and respond to fraudulent activity in real-time. Manual
review processes are too slow, and existing systems often lack the flexibility to
combine structured compliance data with unstructured behavioral analytics. Fraud
Sentinel addresses this by building an integrated fraud detection platform that
scores every transaction in real-time using a hybrid rule-engine + ML approach,
stores structured data in PostgreSQL and enriched scoring context in MongoDB, and
provides analysts with a comprehensive dashboard for alerts, investigations, case
management, and regulatory (SAR) filing — all backed by Redis-powered caching
for low-latency performance.

3. OBJECTIVES
================================================================================

   tables covering users, merchants, transactions, alerts, cases, case notes,
   fraud rules, and SARs — supporting ACID-compliant operations.

   scoring details, capturing the full fraud engine context (risk signals,
   triggered rules, ML anomaly scores) for audit and analytics.

   sub-100ms API response times and real-time per-user velocity breach detection.

   configurable rules (amount thresholds, velocity checks, geolocation anomalies,
   device fingerprinting, ML anomaly detection) and produces automated
   approve/flag/block/challenge decisions.

5. Develop RESTful APIs (Express.js) for full CRUD operations across all
   entities, powering a modern React frontend dashboard.

6. Create a comprehensive analyst dashboard with 12+ views including transaction
   monitoring, alert triage, case management, SAR filing, analytics, and ML
   model performance metrics.

5. DATABASE DESIGN SUMMARY
================================================================================

--------------------------------------------------------------------------------
5.1 SQL Tables Created (PostgreSQL via Drizzle ORM)
--------------------------------------------------------------------------------

1. users         — Stores user account profiles: name, email, phone, risk score,
                   risk tier, transaction counts, fraud counts, total spend,
                   card count, and activity/block/flag timestamps.

2. merchants     — Stores merchant profiles: name, category, country, status
                   (active/blocked/under_review), fraud rate, transaction count,
                   and block reason/timestamp.

3. transactions  — Core transaction records: card/user/merchant IDs, amount,
                   currency, status (approved/flagged/blocked), risk score &
                   level, fraud type, geolocation (country/city/lat/lng), IP,
                   device fingerprint, response time, triggered rules, anomaly
                   score, and review fields.

4. alerts        — System-generated alerts linked to transactions and users:
                   type, severity (info/warning/high/critical), status
                   (open/acknowledged/resolved), title, description, rule ref.

5. cases         — Investigation case management: case number, title, description,
                   status (open/in_progress/escalated/closed), priority, fraud
                   type, estimated/confirmed loss, assignee, linked transaction
                   and alert IDs, SAR reference.

6. case_notes    — Investigator notes attached to cases: author, content, and
                   timestamp for building an investigation audit trail.

7. rules         — Configurable fraud detection rules: name, description, type,
                   enabled/disabled, priority, action (flag/block/alert/monitor),
                   JSONB conditions, triggered count, and false positive rate.

8. sar           — Suspicious Activity Reports for regulatory compliance: SAR
                   number, case reference, subject details, reporting officer,
                   filing institution, activity type, description, amount
                   involved, date range, submission status.


--------------------------------------------------------------------------------
5.2 MongoDB Collections Created
--------------------------------------------------------------------------------

Database: fraud_detection

1. transaction_details — Stores the full enriched context of every scored
   transaction including the original request payload (ctx: cardId, userId,
   merchantId, amount, currency, location, deviceFingerprint), computed
   riskScore (0-100), riskLevel, decision (approved/flagged/blocked/
   challenge_2fa), triggeredRules array, anomalyScore (0.0-1.0 from ML model),
   and creation timestamp. Written asynchronously after each fraud engine
   evaluation for audit and analytics.

   Indexes:
   - { transactionId: 1 }       unique, for cross-referencing with PostgreSQL
   - { "ctx.userId": 1 }        for per-user lookup queries
   - { createdAt: -1 }          for time-series analytics
   - { decision: 1 }            for filtering blocked/flagged transactions

5.3 Redis Cache Keys (Upstash)
--------------------------------------------------------------------------------

Key Pattern              TTL      Description
-----------------------  -------  ----------------------------------------------

trends:{period}          300s     Fraud trend data (byDay, byFraudType, byCountry)
transactions:{query}     5s       Paginated transaction list cache
velocity:{userId}        3600s    Per-user tx count in rolling 1-hour window


--------------------------------------------------------------------------------
5.3 How SQL, MongoDB, and Redis Work Together
--------------------------------------------------------------------------------

PostgreSQL serves as the system of record for all structured, compliance-critical
data — user profiles, merchant data, transaction records, alerts, investigation
cases, fraud rules, and SARs. It provides ACID compliance and relational joins.

MongoDB acts as an audit and analytics store for enriched fraud scoring data.
After each transaction is scored by the fraud engine, the full scoring context
is written asynchronously to the transaction_details collection. The
transactionId field links each MongoDB document back to transactions.id in
PostgreSQL.

Redis provides a high-performance caching and real-time velocity tracking layer.
API responses are cached with short TTLs (5-300 seconds). Redis also tracks
per-user transaction velocity (velocity:{userId}) with a 1-hour rolling window —
the fraud engine reads this counter to detect velocity breach attacks in
real-time (>10 txns/hour = blocked, >5 = flagged).

   PostgreSQL  <--- transactionId --->  MongoDB (transaction_details)
        |
        |--- cache + velocity:{userId} --->  Redis (Upstash)

6. SAMPLE SQL QUERIES (5 Examples)
================================================================================

Query 1: Get today's dashboard summary statistics
----------------------------------------------------------------------
SELECT count(*) AS total_count, sum(amount) AS total_amount
FROM transactions
WHERE created_at >= CURRENT_DATE;

SELECT count(*) AS fraud_count, sum(amount) AS fraud_amount
FROM transactions
WHERE created_at >= CURRENT_DATE AND status = 'blocked';

SELECT count(*) AS critical_open
FROM alerts
WHERE status = 'open' AND severity = 'critical';

SELECT avg(risk_score) AS avg_risk, avg(response_time) AS avg_response
FROM transactions
WHERE created_at >= CURRENT_DATE;


Query 2: Retrieve paginated transactions with multi-filter support
----------------------------------------------------------------------
SELECT *
FROM transactions
WHERE status = 'flagged'
  AND risk_level = 'high'
  AND created_at >= '2026-05-01T00:00:00Z'
  AND created_at <= '2026-05-08T23:59:59Z'
ORDER BY created_at DESC
LIMIT 50 OFFSET 0;


Query 3: Get top 10 highest-risk users with fraud statistics
----------------------------------------------------------------------
SELECT id, name, email, phone, risk_score, risk_tier,
       fraudulent_transactions, total_spend, blocked_at, flagged_at
FROM users
WHERE risk_score > 50
ORDER BY risk_score DESC
LIMIT 10;


Query 4: Alert breakdown grouped by severity, type, and status
----------------------------------------------------------------------
SELECT severity, count(*) AS count FROM alerts GROUP BY severity;
SELECT type, count(*) AS count FROM alerts GROUP BY type;
SELECT status, count(*) AS count FROM alerts GROUP BY status;


Query 5: Transaction velocity analysis over 24 hours
----------------------------------------------------------------------
SELECT
  date_trunc('hour', created_at) AS bucket,
  count(*) AS total,
  count(*) FILTER (WHERE status = 'blocked') AS blocked,
  sum(amount) AS total_amount
FROM transactions
WHERE created_at >= NOW() - INTERVAL '24 hours'
GROUP BY bucket
ORDER BY bucket;

7. SAMPLE MONGODB QUERIES (5 Examples)
================================================================================

Query 1: Find all blocked transactions for a specific user
----------------------------------------------------------------------
db.transaction_details.find({
  "ctx.userId": "user-sim-001",
  decision: "blocked"
}).sort({ createdAt: -1 });


Query 2: Aggregate fraud statistics by country
----------------------------------------------------------------------
db.transaction_details.aggregate([
  { $match: { decision: { $in: ["blocked", "flagged"] } } },
  { $group: {
      _id: "$ctx.location.country",
      totalFraud: { $sum: 1 },
      totalAmount: { $sum: "$ctx.amount" },
      avgRiskScore: { $avg: "$riskScore" }
  }},
  { $sort: { totalFraud: -1 } },
  { $limit: 10 }
]);


Query 3: Find transactions where a specific rule was triggered
----------------------------------------------------------------------
db.transaction_details.find({
  triggeredRules: "HIGH_AMOUNT_THRESHOLD",
  riskScore: { $gte: 70 }
}).projection({
  transactionId: 1,
  "ctx.merchantName": 1,
  "ctx.amount": 1,
  riskScore: 1,
  decision: 1,
  createdAt: 1
}).sort({ createdAt: -1 });


Query 4: Get high anomaly-score transactions from the last 24 hours
----------------------------------------------------------------------
db.transaction_details.find({
  anomalyScore: { $gte: 0.3 },
  createdAt: { $gte: new Date(Date.now() - 24 * 60 * 60 * 1000) }
}).sort({ anomalyScore: -1 });


Query 5: Aggregate triggered rules frequency and avg risk score per rule
----------------------------------------------------------------------
db.transaction_details.aggregate([
  { $unwind: "$triggeredRules" },
  { $group: {
      _id: "$triggeredRules",
      count: { $sum: 1 },
      avgRiskScore: { $avg: "$riskScore" },
      avgAmount: { $avg: "$ctx.amount" },
      blockedCount: {
        $sum: { $cond: [{ $eq: ["$decision", "blocked"] }, 1, 0] }
      }
  }},
  { $sort: { count: -1 } }
]);

                                           writes asynchronously (non-blocking).

Real-time velocity detection without       Implemented Redis as an in-memory
high latency                               counter for per-user transaction
                                           velocity with 1-hour TTL, enabling
                                           O(1) velocity checks.

Handling JSONB conditions for flexible     Used PostgreSQL's native JSONB column

                                           conditions without schema changes.

                                           4 workspace packages.

13. CONCLUSION
================================================================================

Through the development of Fraud Sentinel, we gained deep hands-on experience
with designing and integrating a polyglot database architecture — combining
PostgreSQL for structured compliance data, MongoDB for flexible audit/analytics,
and Redis for real-time caching and velocity tracking. The project demonstrated
how each database technology serves a distinct purpose: PostgreSQL provides ACID
guarantees and relational integrity for financial records; MongoDB offers
schema-flexible document storage for enriched scoring context; and Redis delivers
sub-millisecond in-memory operations for velocity breach detection. Building the
fraud engine reinforced the importance of rule-based + ML hybrid approaches in
real-world fraud detection. The full-stack implementation — from Drizzle ORM
schemas to Express APIs to a 12-page React dashboard — provided comprehensive
experience with modern web development, monorepo management, and cloud-hosted
database services (Neon, MongoDB Atlas, Upstash). Fraud Sentinel successfully
achieves its goal of real-time fraud detection and lifecycle management,
processing transactions in under 100ms while providing analysts with a rich,
interactive investigation platform.
